# 자기 설명적 작성 원칙

## 적용 범위

- **CHANGELOG.md**: 사용자가 체감하는 기능 변화 중심으로 서술
- **커밋 메시지**: 기능 변화 중심이되, 기술적 구현 방법 병기 허용 (개발자가 주 독자)
- 사용자에게 보이는 모든 텍스트

## 규칙: 내부 용어 → 기능 설명으로 변환

프로젝트 내부에서만 통용되는 용어를 사용자 대면 텍스트에 사용하지 않는다.
내부 용어를 발견하면, 해당 변경이 가져오는 **기능적 결과**로 대체한다.

**내부 용어 예시**:
- `Phase 0`, `Phase 1`, `Phase 3G`, `Phase 3I`, `Phase 3A`, `Phase 3S`, `Phase 4`
- `RC-1`, `RC-2` 등 내부 추적 ID
- `Step 1 — Pruning`, `Step 2 — Progressive Disclosure` 등 내부 워크플로우 단계명

## 원칙: 기능 중심 서술

내부 구현 위치가 아닌, **사용자가 체감하는 기능 변화**를 기술한다.

### CHANGELOG (사용자 대상 — 가치 중심)

**❌ 나쁜 예**:
```
- Phase 0에 settings.json 로딩 단계 추가
- Phase 4에 백업 메커니즘 추가
```

**✅ 좋은 예**:
```
- 커맨드 실행 시 `.claude/settings.json` 설정 자동 로딩
- improve/sync 모드에서 기존 CLAUDE.md 자동 백업
```

### 커밋 메시지 (개발자 대상 — 기술 컨텍스트 허용)

**❌ 나쁜 예**:
```
Phase 0에 settings.json 로딩 단계 추가
```

**✅ 좋은 예**:
```
초기화 시 `.claude/settings.json` 파싱하여 사용자 설정 로드
```

## 기술 용어 처리

- 업계 표준 용어 (`API`, `UI`, `JSON`, `CRUD` 등)는 그대로 사용 가능
- 프로젝트 고유 기술 용어는 행동/결과 위주로 풀어서 서술
  - ❌ 디바운스 로직 추가
  - ✅ 중복 입력 방지 처리 추가 (CHANGELOG) / 입력 디바운스 처리 추가 (커밋)

## 판단 기준
